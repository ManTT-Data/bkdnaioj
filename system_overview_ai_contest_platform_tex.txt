\section{Introduction}

This document presents a high-level description of the proposed web-based platform for organizing, evaluating, and practicing student AI competitions. The goal of the platform is not only to receive submissions and return scores, but also to support the full operational workflow of a real AI contest, including contest setup, multiple tasks, multiple evaluation phases, team-based or individual participation, scoreboards, virtual participation, practice mode, clarifications, technical tickets, and rejudging.

Unlike a traditional programming contest system, this platform is designed for AI competition workflows in which contestants typically submit artifacts such as prediction files or ZIP packages rather than plain source code only. The platform then evaluates those artifacts on the server side by using predefined judge scripts for each phase.

\section{Core Idea of the System}

The core principle of the platform is simple:

\begin{quote}
Contestants submit artifacts in a required format, while the server is responsible for validating the submission, reading the correct public or private reference data, running the corresponding judge script, computing the score, and updating the relevant leaderboard.
\end{quote}

Therefore, contestants do not directly influence the official execution environment used for evaluation. This design improves consistency, fairness, reproducibility, and operational control.

\section{Main Goals}

The platform is intended to satisfy the following goals:

\begin{itemize}
    \item support AI competitions with multiple tasks and multiple phases;
    \item support both individual and team participation;
    \item support official contests, virtual contests, and practice mode;
    \item support artifact-based submission, especially ZIP-based submission workflows;
    \item allow each phase to have its own evaluation logic through a judge script;
    \item provide public-facing scoreboards while keeping evaluation logic on the server;
    \item support contest operations such as announcements, clarifications, and technical tickets;
    \item support later extensions such as model verification, rerunning, and managed execution.
\end{itemize}

\section{System Scope}

The initial version of the platform focuses on the essential contest workflow:

\begin{itemize}
    \item user accounts and team management;
    \item contest creation and configuration;
    \item task and phase definition;
    \item submission upload and validation;
    \item server-side judging using phase-specific scripts;
    \item score computation and leaderboard updates;
    \item official, virtual, and practice participation modes;
    \item communication between contestants and organizers.
\end{itemize}

The first version does \textbf{not} aim to provide a full cloud coding workspace, browser-based notebook training environment, or unrestricted custom evaluator upload by all organizers.

\section{Users and Roles}

The platform contains three main categories of actors.

\subsection{Contestants}
Contestants use the platform to:
\begin{itemize}
    \item register and log in;
    \item create or join teams;
    \item register for contests;
    \item read task statements and phase rules;
    \item upload submissions;
    \item monitor submission status;
    \item view scoreboards;
    \item ask clarification questions;
    \item report technical issues.
\end{itemize}

\subsection{Organizers / Jury / Administrators}
Organizers use the platform to:
\begin{itemize}
    \item create contests, tasks, and phases;
    \item configure submission rules and visibility;
    \item assign judge scripts to phases;
    \item monitor submission queues and evaluation results;
    \item publish announcements;
    \item answer clarification requests;
    \item resolve technical tickets;
    \item trigger rejudging when needed.
\end{itemize}

\subsection{Judge Workers}
Judge workers are backend execution units that:
\begin{itemize}
    \item receive validation and judging jobs from a queue;
    \item inspect and validate uploaded artifacts;
    \item run the correct judge script for the target phase;
    \item compute scores and return evaluation results;
    \item record logs for debugging and traceability.
\end{itemize}

\section{Contest Structure}

\subsection{Contest}
A contest is the top-level object. It defines global information such as title, description, registration period, official start and end time, entry policy, and general rules.

\subsection{Task}
A contest contains one or more tasks. Each task corresponds to one problem or challenge in the competition. A task has a title, a statement, and a submission schema describing what contestants are expected to upload.

\subsection{Phase}
Each task contains multiple phases. In the target design, a task typically uses four phases:

\begin{enumerate}
    \item \textbf{Public Test}
    \item \textbf{Final Public}
    \item \textbf{Private Test}
    \item \textbf{Final Private}
\end{enumerate}

These four phases represent different evaluation contexts. The exact meaning depends on the contest, but the common interpretation is:

\begin{itemize}
    \item \textbf{Public Test}: fast feedback on public data.
    \item \textbf{Final Public}: a public-phase lock or final public submission checkpoint.
    \item \textbf{Private Test}: evaluation on private data.
    \item \textbf{Final Private}: final official evaluation, possibly with stronger checks.
\end{itemize}

\subsection{Why phases matter}
Phases are not just UI tabs. They define the operational behavior of evaluation:

\begin{itemize}
    \item which artifact format is expected;
    \item whether official, virtual, or practice submission is allowed;
    \item how many submissions are permitted;
    \item which judge script is executed;
    \item whether the scoreboard is shown or frozen.
\end{itemize}

\section{Participation Model}

\subsection{Individual and Team Entries}
The platform supports both individuals and teams.

A global team can exist independently of any contest. When a contest starts, the actual participating object is not the raw team itself but a \textbf{contest entry}. A contest entry may represent:

\begin{itemize}
    \item one individual contestant, or
    \item one team participating in that specific contest.
\end{itemize}

\subsection{Lineup Selection}
A team may contain many members globally, but only a subset of them may participate in a given contest. Therefore, each contest entry maintains its own lineup.

For example, a team may have five global members, while only three of them are selected to participate in one particular contest. This design also supports cases where one user belongs to multiple teams globally but is selected into only one contest entry for a specific event.

\subsection{Official, Virtual, and Practice Modes}
Each contest entry belongs to one of three participation modes:

\begin{itemize}
    \item \textbf{Official}: the real competition instance used for official ranking.
    \item \textbf{Virtual}: a replay instance started by a contestant after the original contest, similar to Codeforces virtual participation.
    \item \textbf{Practice}: open-ended practice mode after a contest has ended.
\end{itemize}

\section{Submission Model}

\subsection{What contestants submit}
In the first version, contestants do not code directly inside the platform. Instead, they prepare artifacts externally and upload them to the system.

Typical submission artifacts include:
\begin{itemize}
    \item a prediction file such as \texttt{.csv} or \texttt{.txt};
    \item a ZIP archive containing required output files;
    \item in more advanced settings, a ZIP archive containing outputs, scripts, checkpoints, or supporting files.
\end{itemize}

\subsection{Why ZIP submissions are natural}
Many AI contest tasks require a structured collection of files. A ZIP-based submission is therefore a natural choice. It allows contestants to submit a package that follows a strict schema defined by the contest organizer.

Examples of such schema requirements include:
\begin{itemize}
    \item required filenames;
    \item required subdirectories;
    \item required text format or column layout;
    \item required model-related files for selected phases.
\end{itemize}

\subsection{Submission lifecycle}
A submission goes through the following lifecycle:

\begin{enumerate}
    \item the contestant uploads the artifact;
    \item the system stores it in object storage;
    \item a validation job is created;
    \item a judge worker inspects the artifact;
    \item if necessary, the archive is extracted;
    \item the structure is checked against the expected schema;
    \item if validation fails, the submission is marked as failed with an error message;
    \item if validation succeeds, the corresponding judge script is executed;
    \item the script computes the score;
    \item the platform stores the score and updates the relevant leaderboard.
\end{enumerate}

\section{Judge Scripts}

\subsection{Definition}
Each phase is associated with a dedicated \textbf{judge script}. This is the key execution component of the evaluation pipeline.

A judge script is responsible for:
\begin{itemize}
    \item reading the submitted artifact;
    \item extracting ZIP archives if needed;
    \item checking the required files and structure;
    \item reading the corresponding reference data for that phase;
    \item computing the final score;
    \item returning a standardized result to the platform.
\end{itemize}

\subsection{Design philosophy}
Instead of building an overly abstract metrics engine in the first version, the platform uses a practical phase-based approach:

\begin{quote}
Each phase has its own judge script, and that script knows exactly how to validate and score submissions for that phase.
\end{quote}

This is conceptually similar to special judges or custom checkers in algorithmic contest systems, except that the artifact and scoring logic may be more complex for AI competitions.

\subsection{Why this design is appropriate}
This approach is simple and robust because it matches how real contest statements are written. If a phase requires a ZIP package with a specific layout, the judge script can directly enforce that layout. If a phase should evaluate against public data, it reads the public reference data. If another phase must evaluate against private data, it reads the private reference data. If something is missing or malformed, it returns an explicit validation error.

\section{Scoring Model}

\subsection{Single visible score per submission}
The first version of the platform follows the same public-facing philosophy used by systems such as Kaggle and many programming contests:

\begin{quote}
Each submission produces one final visible score for ranking purposes.
\end{quote}

The system may internally record auxiliary metadata or breakdown information, but the user-facing scoreboard mainly needs:
\begin{itemize}
    \item rank,
    \item contestant or team name,
    \item score,
    \item number of submissions,
    \item latest submission time.
\end{itemize}

\subsection{Score computation}
The exact score is computed by the phase-specific judge script. Therefore, the platform does not need to hardcode a universal metrics framework in the initial version. The score is whatever the phase judge defines as the official scalar value for that phase.

\section{Leaderboard Model}

\subsection{Task-Phase Leaderboards}
For each task-phase pair, the platform maintains a leaderboard that ranks entries based on the selected rule (for example, best submission or latest submission).

Examples:
\begin{itemize}
    \item Task 1 -- Public Test
    \item Task 1 -- Private Test
    \item Task 2 -- Final Public
    \item Task 2 -- Final Private
\end{itemize}

\subsection{Contest-Phase Leaderboards}
In addition to per-task boards, the platform also provides contest-level leaderboards for each contest phase. These are the four contest-wide tabs that summarize the total score across tasks for:

\begin{itemize}
    \item Public Test
    \item Final Public
    \item Private Test
    \item Final Private
\end{itemize}

\subsection{Mode-based filtering}
Leaderboards can also be filtered by participation mode:
\begin{itemize}
    \item Official
    \item Virtual
    \item Practice
    \item All
\end{itemize}

However, only the official leaderboard should be treated as the formal competition ranking.

\section{Virtual and Practice Workflow}

\subsection{Virtual contests}
A virtual contest is a replay of a finished official contest. A contestant can start a virtual session at any chosen time. The system then creates a virtual contest entry and computes the phase timing relative to that entry's personal start time.

Virtual participation is useful because it allows users to practice under contest-like timing conditions without affecting the official ranking.

\subsection{Practice mode after contest}
Once a contest is over, the platform may allow users to view tasks and submit again in practice mode. The practical design choice for the first version is:

\begin{itemize}
    \item allow practice on the public and private test phases;
    \item do not reopen the final phases that require heavier verification or model-related packages.
\end{itemize}

This reduces system load while still supporting meaningful practice.

\section{Communication and Contest Operations}

A real contest platform needs more than submission and scoring. The system therefore includes operational communication modules.

\subsection{Announcements}
Organizers can publish:
\begin{itemize}
    \item global contest announcements,
    \item task-specific announcements,
    \item pinned urgent notices.
\end{itemize}

\subsection{Clarifications}
Contestants can ask questions about:
\begin{itemize}
    \item task statements,
    \item rules,
    \item ambiguous requirements,
    \item expected formats.
\end{itemize}

Organizers can answer privately or make an answer public for all contestants.

\subsection{Technical tickets}
Technical tickets are separate from clarifications. They are used for issues such as:
\begin{itemize}
    \item failed uploads,
    \item unexpected judging failures,
    \item score disputes,
    \item platform-side technical problems.
\end{itemize}

\section{Submission History and User Statistics}

The platform keeps submission history for each contest entry. Users can inspect their previous submissions, scores, and statuses.

Because contest entries have explicit lineups, any team submission naturally belongs to all members of that lineup for participation statistics.

In future or extended versions, the platform can derive user-centric statistics such as:
\begin{itemize}
    \item number of official contests participated in;
    \item number of virtual contests participated in;
    \item number of tasks completed;
    \item number of top-10, top-3, top-2, and top-1 finishes.
\end{itemize}

\section{System Architecture}

At a high level, the platform consists of four main layers.

\subsection{Web Application Layer}
This layer provides:
\begin{itemize}
    \item contestant-facing pages,
    \item organizer dashboards,
    \item authentication and authorization,
    \item contest and task management,
    \item scoreboard views,
    \item communication interfaces.
\end{itemize}

\subsection{Database Layer}
The database stores structured contest data such as:
\begin{itemize}
    \item users,
    \item teams,
    \item contest entries,
    \item tasks and phases,
    \item submissions,
    \item evaluation jobs,
    \item leaderboard states,
    \item announcements, clarifications, and tickets.
\end{itemize}

\subsection{Object Storage Layer}
This layer stores uploaded artifacts such as:
\begin{itemize}
    \item ZIP files,
    \item prediction files,
    \item logs,
    \item possibly other generated artifacts in future versions.
\end{itemize}

\subsection{Judge Worker Layer}
Judge workers process queued jobs. They perform validation, execute judge scripts, compute scores, and return results to the application.

\section{Technology Direction}

The recommended storage strategy is:
\begin{itemize}
    \item \textbf{PostgreSQL} as the main relational database;
    \item \textbf{Redis} for caching, queue support, and transient operational state;
    \item \textbf{MinIO or S3-compatible object storage} for submission artifacts and large files.
\end{itemize}

The reason for this choice is that the system is strongly relational at its core, while still benefiting from JSON fields and object storage for flexible metadata and uploaded files.

\section{Deliberate Non-Goals for Version 1}

To keep the first version focused and achievable, the following are intentionally excluded:
\begin{itemize}
    \item a full browser-based IDE or notebook environment;
    \item user-controlled training jobs on shared GPU servers;
    \item a general-purpose evaluator marketplace;
    \item unrestricted custom evaluator upload for all organizers;
    \item blockchain-based submission verification;
    \item a full rating system similar to Codeforces.
\end{itemize}

These may be explored later, but they are not part of the initial system scope.

\section{Future Extensions}

The platform is designed so that future versions may extend it in the following directions:
\begin{itemize}
    \item managed remote execution for contestant code;
    \item final-phase verification using model packages or checkpoints;
    \item richer analytics and participant statistics;
    \item rating for official individual contests;
    \item browser-based editing and controlled execution environments;
    \item more advanced audit and reproducibility tooling.
\end{itemize}

\section{Conclusion}

The proposed platform is a contest-oriented AI evaluation system rather than a traditional programming-only online judge. It supports multiple tasks, multiple phases, multiple participation modes, structured artifact submission, server-side phase-specific judging, task-level and contest-level scoreboards, and essential contest operations such as clarifications and technical tickets.

Its most important design choice is practical simplicity: contestants upload structured artifacts, and the platform performs validation and scoring on the server side through dedicated judge scripts for each phase. This makes the system well aligned with real AI competition workflows while keeping the first version technically achievable and operationally clear.
